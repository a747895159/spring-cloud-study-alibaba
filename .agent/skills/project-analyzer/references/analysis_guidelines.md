# 项目分析指南

## 概述

本指南提供了在分析现有项目时应遵循的步骤和最佳实践，旨在确保分析的全面性、准确性和一致性，从而为后续的文档生成和项目理解奠定基础。

## 分析步骤

### 1. 明确分析目标

*   **目的**: 理解项目的核心功能、技术栈、架构模式，为新项目启动或重构提供支持。
*   **产出**: 结构化的项目文档、代码结构分析报告。

### 2. 收集项目信息

*   **代码库**: 获取完整的代码仓库，包括所有分支和历史记录。
*   **现有文档**: 收集所有可用的现有文档，如：`README.md`、设计文档、API 文档、用户手册、Git Commit 历史、Issue 跟踪系统（如果有）。
*   **团队访谈**: 如果可能，与项目相关人员（开发者、产品经理、架构师）进行访谈，了解项目背景、痛点、未来规划等非文档化信息。

### 3. 项目结构概览

*   **目录结构**: 快速浏览项目根目录，识别主要的业务模块、技术分层（如 `src`, `app`, `core`, `utils`, `controllers`, `services`, `models`, `tests`, `docs`）。
*   **配置文件**: 识别 `package.json`, `requirements.txt`, `pom.xml`, `build.gradle`, `Dockerfile`, `config` 目录等，以快速了解技术栈和依赖。

### 4. 代码层面分析

*   **入口点识别**: 找出应用程序的启动文件、Web 服务的路由定义、主要业务逻辑的入口函数。
*   **模块依赖**: 分析模块之间的导入关系，识别核心模块和辅助模块。
*   **代码风格与规范**: 观察项目代码的风格、命名约定、注释情况。
*   **数据库交互**: 识别数据库连接、ORM 配置、数据迁移脚本、SQL 语句等。
*   **API 接口**: 识别 RESTful API、GraphQL 接口的定义及其参数、响应结构。
*   **异步/并发模型**: 识别是否存在异步任务、消息队列、多线程/多进程等机制。
*   **错误处理机制**: 观察项目中的异常捕获、错误日志记录方式。

### 5. 文档层面分析

*   **提取关键信息**: 从现有文档中提取项目名称、描述、目标、功能列表、技术栈线索等。
*   **识别缺失信息**: 记录现有文档中未涵盖但对项目理解至关重要的信息点。

### 6. 功能与交互总结

*   **用户故事/用例**: 尝试从用户视角总结项目提供的核心功能。
*   **关键交互流程**: 梳理用户与系统之间，以及系统内部模块之间的主要交互流程。

### 7. 技术栈识别与评估

*   **编程语言**: 确定主要编程语言及版本。
*   **框架/库**: 识别使用的主要框架（如 Spring Boot, Django, React, Vue）和关键第三方库。
*   **数据库**: 识别使用的数据库类型（如 MySQL, PostgreSQL, MongoDB, Redis）。
*   **部署环境**: 了解常见的部署方式（如 Docker, Kubernetes, Serverless）和 CI/CD 流程。

### 8. 架构模式识别

*   识别项目可能采用的架构模式（如分层架构、微服务、事件驱动、六边形架构）。
*   注意代码结构与架构模式之间的匹配度。

### 9. 数据模型分析

*   识别核心业务实体。
*   尝试从代码（ORM 模型）或数据库 schema 中推断实体关系。

### 10. 代码结构深度分析 (重构考量)

*   **分层识别**: 识别控制器层、服务层、领域层、持久层、基础设施层等。
*   **职责边界**: 评估各层级和模块的职责是否清晰，是否存在职责混淆或过度耦合。
*   **通用组件/库**: 识别项目内部是否创建了可复用的通用组件或库。
*   **设计模式**: 识别项目中使用的设计模式及其应用场景。
*   **代码异味**: 识别潜在的代码异味（如大类、长方法、重复代码），为重构提供依据。

## 输出建议

*   以 Markdown 格式生成结构化文档。
*   包含目录、清晰的标题和子标题。
*   代码片段应使用代码块和语法高亮。
*   图表应使用 PlantUML 或其他文本图工具生成，或提供占位符。
*   确保文档内容准确、详细且面向读者。
