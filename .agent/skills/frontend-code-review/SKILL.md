# 前端代码审查

## 目的
当用户要求审查前端代码（特别是 `.tsx`、`.ts` 或 `.js` 文件）时使用此技能。支持两种审查模式：

1. **待变更审查** – 检查暂存区/工作目录中准备提交的文件，在提交前标记检查清单违规。
2. **文件定向审查** – 审查用户指定的特定文件，并报告相关的检查清单结果。

对每个适用的文件和模式，严格遵循以下检查清单。

## 检查清单
参见 [references/code-quality.md](references/code-quality.md)、[references/performance.md](references/performance.md)、[references/business-logic.md](references/business-logic.md) 获取按类别划分的动态检查清单 — 将其视为需遵循的规范规则集。

为每个规则违规标记紧急程度元数据，以便未来的审查者可以优先处理修复。

## 审查流程
1. 打开相关组件/模块。收集与类名、React Flow Hooks、属性记忆化和样式相关的行。
2. 对于审查要点中的每条规则，记录代码偏差之处并捕获一段代表性代码片段。
3. 按照下面的模板编写审查部分。先按 **紧急** 标志分组，然后按类别顺序（代码质量、性能、业务逻辑）排列。

## 必需的输出
调用时，响应必须严格遵循以下两个模板之一：

### 模板 A（存在问题）
```
# 代码审查
发现 <N> 个紧急问题需要修复：

## 1 <问题简述>
文件路径：<path> 第 <line> 行
<相关代码片段或指针>


### 建议修复
<修复建议简述>

---
...（对每个紧急问题重复）...

发现 <M> 个改进建议：

## 1 <建议简述>
文件路径：<path> 第 <line> 行
<相关代码片段或指针>


### 建议修复
<修复建议简述>

---

...（对每个建议重复）...
```

如果没有紧急问题，省略该部分。如果没有建议，省略该部分。

如果问题数超过 10 个，汇总为"10+ 个紧急问题"或"10+ 个建议"，只输出前 10 个问题。

不要压缩各部分之间的空行；保持原样以提高可读性。

如果使用模板 A（即存在需修复的问题）且至少一个问题需要代码变更，在结构化输出之后追加一个简短的后续问题，询问用户是否希望您应用建议的修复。例如："您是否希望我使用建议修复部分来解决这些问题？"

### 模板 B（无问题）
```
## 代码审查
未发现问题。
```
